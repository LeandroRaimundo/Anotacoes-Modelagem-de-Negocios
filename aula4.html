ANOTAÇÕES DA AULA 4 DE MODELAGEM DE NEGÓCIOS E REQUISITOS

Requisitos Não Funcionais

  • Os requisits de software deste tipo não apresnetam relação direta com serviços específicos fornecidos pelo software aos usuários.
  • Não são menos importantes que os requisitos funcionais.
  • Os requisitos não funcionais geralmente são mais críticos e podem inutilizar o sistema.
  • São requisitos críticos que podem, por exemplo, inutilizar o aplicativo de um banco, prejudicando muitas pessoas.

Formas de Categorizar Requisitos Não Funcionais

  • Requisitos de Produto
    • - Especificam o comportamento do software durante a execução (Como o software precisa funcionar de forma organizada)

  • Requisitos Organizacionais
    • - Requisitos amplos relacionados ao sistema.


    Operacional: Como vai ser utilizado.
    Ex: App de banco para pessoas acessarem suas contas pelo celular.


    Ambiental: Sistema Operacional. plataforma, etc.
    Ex: Sistema Windows, Linux ou IOS.


    De Desenvolvimento: Linguagem de programação, processo de desenvolvimento, etc.
    Ex: O software irá utilizar Python, Cobol, Java e os processos, como, Python será utilizado para fazer cálculos dos usuários.

  • Requisitos Externos
    • - Regulatórios, normativas, aspectos éticos e políticos.

Exemplos de Requisitos Não Funcionais

  • O tempo que o sistema poderá ficar fora do ar não pode ser superior
    a 5 segundos em qualquer dia.

  • Os usuários do sistema devem se registrar utilizando um email e senha.

  • O manual de uso deve estar disponível online.

Sobre o FURPS+

  • O FURPS+ é um sistema para a classificação de requisitos.

    • Funcionality: é ligado aos requisitos funcionais (noção do que precisa fazer).
    • Usability: é o atributo que avalia a interface com o usuário.
    • Reability: Se refere a integridade, conformidade e interopabilidade do software.
    • Performance: Se refere ao tempo de resposta, consumo de memória, utilização da CPU, etc.
    • Supportability: Testabilidade, adaptabilidade, manutenibilidade, compatibilidade, etc.

    SLA - Service Level Agreement

    • Acordo de Nível de Serviço (ANS) ou (SLA) é fundamental para qualquer contrato na área de TI. Trata-se de todos os serviços que o contratante espera do fornecedor.
    • É um documento contratual que deve ser periodicamente revisto.
    • O cumprimento dos níveis de serviço são observados através das métricas

    Documento de Especificação de Requisitos

    • Os documentos de requisitos de software constituem em uma declaração oficial do que o software deve fazer.
    • Especialmente essencial quando o desenvolvimento é de terceiros.
    • Pode não haver um documento de especificação de requisitos quando o projeto é perene, de manutenção constante, para um cliente interno, etc.

    (RF001) para Requisito Funcional e (RNF001) para Requisito Não Funcional.
