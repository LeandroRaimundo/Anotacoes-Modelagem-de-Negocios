Requisitos

- Antes de iniciar qualquer trabalho técnico, é uma boa ideia criar um conjunto de requisitos para todas as tarefas de engenharia

Quem realiza?

  • Engenheiro de Software
  • Analistas de Sistemas
  • Scrum Master
  • Product Owner (PO)
  • E outras pessoas envolvidas no projeto de Software

Engenharia de Requisitos

  • ER é o conjunto de tarefas e técnicas que conduzem a um entendimento dos requisitos.
  • O projeto precisa de um modelo para os requisitos funcionarem.
  • A empresa define como o projeto deve ser feito. Ex. Deve ter logo da empresa, um gabarito, sistema de login, etc.

- O grupo precisa de um líder para marcar reuniões e(ou) planejar.

- Durante a reunião, o líder precisa documentar ela, com dia, horário, assunto, se houve faltas, etc.

Funções da ER

Constrói uma ponte entre o projeto e a construção do projeto,

Define as necessidades do negócio (Objetivo do Projeto),

Descrição de cenários de usuários,

Determinação das funções e recursos (definir o que cada parte vai ser responável),

Identificação das restrições de projeto (O que será possivel fazer ao alcance da equipe),

A computação onipresente permite que a tecnologia esteja presente em diversos objetos do dia a dia, como em máquinas de lavar, ar-condicionados, geladeiras, microondas, etc.

As Tarefas da Engenharia de Requisitos

  1. Concepção - Estabelecer um entendimento sobre o projeto.

  2. Levantamento - Fazer perguntas para o cliente, usuários e aos demais envolvidos para identificar qual a meta da organização em relação ao uso futuro do sistema

  3. Elaboração - Refinar as informações coletadas junto ao cliente. Descrição de como o usuário irá interagir com o software e produção de diagramas suplementares (Documentar o que enxergou para suprir os requisitos)

  4. Negociação - Conciliar conflitos por meio de negociações. Ususários e clientes devem participar da prioriação, avaliação de custos e riscos, os requisitos serão alterados para tentar satisfazer à todos

  5. Especificação - Documentar as Especificações de requisitos, onde constam:
    - Finalidade do produto ou software,
    - Ambiente Operacional (se vai funcionar em Windows, Linus, Android, IOS, etc),
    - Restrições do Projeto,
    - Características do sistema,
    - Requisitos funcionais e não funcionais.

  6. Validação - Etapa de validação irá validar o documento de especificação para certificar que tudo tenha sido declarado de forma não ambígua

  7. Gestão - Examinar cada requisito em relação a um conjunto de perguntas contidas em uma lista de controle.
    Exemplo:
    - Os requisitos estão expressos de forma clara? eles podem ser mal interpretados?
    - A fonte (por exemplo, uma pessoa, uma regulamentação, um documento) do requisito foi identificada?
    - O requisito está limitado em termos quantitativos?

Problemas em Projetos de Software

O Standish Group fez uma análise sobre os fatores críticos para o sucesso dos projetos de software e perceberam que os maiores problemas são relacionadas às atividades de requisitos:

  • Requisitos Incompletos
  • Falta de Envolvimento dos Usuários
  • Mudança de Requisitos e Especificações

Requisitos Funcionais

  • São declarações dos serviços que o sistema deve fornecer, do modo que o sistema deve reagir a determinadas entradas e como se comporta em certas situações.
  • Descrevem a funcionalidade ou serviços que espera que o sistema tenha.

Exemplos:

  1. Um usuário deve poder fazer uma busca na lista de consulta de todas as clínicas.

  2. O sistema deve gerar uma lista de pacientes que estão agendados para consulta naquele dia.

  3. Cadastras clientes

  4. Emitir nota fiscal

Diretrizes para a Redação de Requisitos de Software

  • Definir um formato padrão e garantir que todos os requisitos sejam seguidos.
  • Usar a linguagem coerente para distinguir requisitos obrigatórios e desejáveis.
  • Não supor que os leitores compreendam a linguagem da engenharia de software.
